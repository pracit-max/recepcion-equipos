DOCUMENTACION GENERAL DEL PROYECTO
Sistema de Recepcion y Devolucion de Equipos - Innova Schools

============================================================
INTRODUCCION
============================================================

Este documento explica de forma general y paso a paso el desarrollo del
proyecto "Sistema de Recepcion y Devolucion de Equipos". La documentacion
esta escrita como una historia del proceso: desde la idea inicial, el primer
dia de construccion, las primeras pantallas, las mejoras realizadas, los
cambios importantes, lo que se elimino y las razones de cada decision.

El proyecto nacio como una necesidad de controlar mejor el prestamo de
equipos tecnologicos en las sedes de Innova Schools. Antes, este proceso
podia depender de registros manuales, mensajes, hojas separadas o controles
menos centralizados. Por eso se desarrollo una herramienta web que permitiera
registrar la entrega, controlar la disponibilidad, guardar evidencias y cerrar
la devolucion de cada solicitud.

============================================================
OBJETIVO GENERAL
============================================================

Crear un sistema web que permita registrar, consultar y cerrar prestamos de
equipos tecnologicos por sede, guardando la informacion en Google Sheets y
generando evidencia documental mediante PDFs.

El sistema busca responder estas preguntas:

- Quien recibio el equipo.
- Que equipo o carro recibio.
- En que sede se hizo el prestamo.
- A que hora debe devolverlo.
- Si el equipo tenia alguna novedad.
- Si el equipo fue devuelto correctamente.
- Que solicitudes siguen abiertas.
- Que solicitudes ya fueron cerradas.
- Donde esta el acta o PDF de soporte.

============================================================
PRIMER DIA: NACIMIENTO DEL PROYECTO
============================================================

El primer dia se definio la necesidad principal: crear una plataforma sencilla
para registrar la recepcion de equipos tecnologicos. En esta etapa el sistema
todavia no era una herramienta completa de gestion, sino un formulario para
capturar informacion basica.

Lo primero que se hizo fue organizar la estructura inicial del proyecto:

- Una pagina principal.
- Una carpeta para vistas o formularios.
- Una carpeta para estilos CSS.
- Una carpeta para JavaScript.
- Una carpeta para imagenes.
- Una carpeta para el backend de Google Apps Script.

La primera version se enfoco en que el usuario pudiera ingresar datos como
nombre, cedula, correo, equipo recibido, fecha, hora y firma. El objetivo era
pasar de un registro manual a un formulario digital.

Desde el comienzo se tomo la decision de usar HTML, CSS y JavaScript porque
el sistema debia poder abrirse facilmente en un navegador, sin necesidad de
instalar programas complejos. Tambien se eligio Google Apps Script porque
permitia conectar el formulario con Google Sheets, Google Drive y Gmail.

Logro del primer dia:

- Se definio la idea del sistema.
- Se creo la estructura base del proyecto.
- Se empezo a construir el formulario principal.
- Se definio que Google Sheets seria la base de datos.
- Se empezo a pensar en una solucion sencilla, accesible y facil de mantener.

Mejora lograda:

El sistema dejo de ser un formulario unico y paso a organizarse por sedes.

OCTAVA ETAPA: VERIFICACION DE CORREO INSTITUCIONAL
============================================================

Se agrego el archivo:

- js/verify.js

Este archivo implementa la verificacion del correo institucional. El sistema
solo acepta correos con dominio:

@innovaschools.edu.co

Proceso:

1. El usuario escribe su correo.
2. El sistema valida que sea institucional.
3. Se genera un codigo de 6 digitos.
4. El codigo se envia por correo.
5. El usuario ingresa el codigo.
6. Si el codigo coincide, el correo queda verificado.

Se agregaron tambien:

- Tiempo de expiracion del codigo.
- Limite de intentos.
- Opcion de reenviar codigo.
- Modal para ingresar el codigo.

Por que se hizo:

Esto mejora la seguridad y evita que cualquier persona registre solicitudes
con correos falsos o externos.

Mejora lograda:

El sistema empezo a validar identidad basica mediante correo institucional.

Se agregaron campos como:

- id_solicitud.
- estado.
- hora_entrega.

Estados principales:

- EN_CURSO.
- CERRADO.

Por que se hizo:

Era necesario saber que equipos seguian prestados y cuales ya habian sido
devueltos.

Mejora lograda:

ENTREGA -> USO -> DEVOLUCION -> CIERRE

============================================================
DECIMA CUARTA ETAPA: CIERRE DE SOLICITUDES
============================================================

Se implemento la funcion de cierre de solicitud en frontend y backend.

En frontend:

- cerrarSolicitudFrontend()

En backend:

- cerrarSolicitud(data)

Proceso:

1. Soporte selecciona una solicitud en curso.
2. El sistema carga sus datos.
3. Soporte registra la devolucion.
4. Se valida cantidad devuelta y estado final.
5. Si hay novedad, se exige descripcion y foto.
6. Se captura firma de devolucion.
7. Se envia la informacion al backend.
8. El backend busca la fila por id_solicitud.
9. Se actualiza la misma fila.
10. El estado cambia a CERRADO.

Por que se hizo asi:

Se decidio actualizar la misma fila y no crear una nueva porque asi toda la
historia de una solicitud queda en un unico registro. Esto facilita auditoria,
consulta y generacion de reportes.

Mejora lograda:

La informacion de entrega y devolucion queda unificada.

- Hora maxima de devolucion.
- Hora real de devolucion.
- Tiempo de uso.
- Estado final.
- Novedades.
- Foto si aplica.
- Firma de recepcion.
- Firma de devolucion.
- Conclusion automatica.

Ejemplos de conclusion:

- Equipos devueltos correctamente.
- Se presentaron novedades en la devolucion.
- Devolucion incompleta o con diferencias.

Por que se hizo:

El PDF final permite dejar evidencia completa del ciclo de vida de la
solicitud, no solo de la entrega.

Mejora lograda:

El sistema genera documentacion formal tanto al prestar como al devolver.

============================================================
COSAS QUE SE QUITARON O SIMPLIFICARON
============================================================

1. Campos innecesarios en la vista de registros

Se quitaron campos como:

- Firma digital visible en algunas vistas.
- Cargador.
- Autorizacion.
- Novedad en lugares donde no era necesaria.
- Boton de PDF en ciertos modales.

Por que se quito:

La vista de registros tenia demasiada informacion y podia confundir al usuario.
Se simplifico para mostrar solo lo mas importante.

Resultado:

La consulta quedo mas limpia y facil de leer.

------------------------------------------------------------

2. Boton de descargar PDF en modal de detalle

Se elimino el boton de descargar PDF del modal de detalle en algunas vistas.

Por que se quito:

No siempre era necesario descargar desde ahi y podia duplicar acciones. Se
priorizo una vista mas simple y enfocada.

Resultado:

La interfaz quedo menos cargada.

------------------------------------------------------------

3. Mensajes con emojis

Se quitaron emojis de algunos mensajes, especialmente en avisos de carro no
disponible.

Por que se quito:

Para mantener un tono mas institucional y profesional.

Resultado:

Los mensajes quedaron mas serios y claros.

------------------------------------------------------------

4. Creacion de nueva fila al cerrar solicitud

Se evito crear una nueva fila cuando se cierra una solicitud.

Por que se quito esa idea:

Crear otra fila separaria la entrega de la devolucion y complicaria la
trazabilidad.

Resultado:

Ahora se actualiza la misma fila usando id_solicitud.

------------------------------------------------------------

5. Vista demasiado cargada de registros

Se redujo informacion secundaria en algunas tarjetas y modales.

Por que se hizo:

El usuario necesita consultar rapido nombre, sede, equipo, hora y estado. Si
se muestran demasiados datos al mismo tiempo, la consulta se vuelve lenta.

Resultado:

La visualizacion quedo mas enfocada en lo importante.

QUE CAMBIO EN LA IDEA ORIGINAL
============================================================

La idea original era hacer un formulario para registrar la entrega de equipos.
Con el avance del proyecto, esa idea crecio.

Antes:

Formulario de recepcion.

Despues:

Sistema completo de gestion de prestamos.

El sistema ahora incluye:

- Recepcion.
- Validacion.
- Disponibilidad.
- Evidencia.
- Seguimiento.
- Devolucion.
- Cierre.
- Actas PDF.
- Control por sede.
- Gestion de equipos.

Este cambio fue importante porque un prestamo no termina cuando se entrega el
equipo. Termina cuando el equipo vuelve y se confirma su estado.

POR QUE ALGUNAS DECISIONES FUERON IMPORTANTES
============================================================

Usar Google Sheets:

Porque permite guardar informacion de forma sencilla, visible y facil de
consultar.

Usar Google Apps Script:

Porque conecta directamente con Sheets, Drive y Gmail sin crear un servidor
externo.

Crear formularios por sede:

Porque reduce errores y facilita el uso por colegio.

Agregar id_solicitud:

Porque permite identificar cada prestamo como un caso unico.

Actualizar la misma fila:

Porque mantiene entrega y devolucion en un solo registro.

Agregar firma digital:

Porque reemplaza evidencia fisica.

Agregar PDF final:

Porque documenta todo el ciclo del prestamo.

Quitar elementos innecesarios:

Porque una interfaz mas limpia permite trabajar mas rapido y con menos errores.

============================================================
CONCLUSION GENERAL
============================================================

El proyecto empezo como un formulario de recepcion de equipos y evoluciono
hasta convertirse en un sistema completo para gestionar prestamos tecnologicos.

Durante el desarrollo se construyeron paginas por sede, se agrego diseno
responsive, se implemento logica en JavaScript, se conecto con Google Apps
Script, se guardo informacion en Google Sheets y se generaron documentos PDF
en Google Drive.

Una de las mejoras mas importantes fue agregar el modulo de devoluciones,
porque permitio cerrar el ciclo completo del prestamo. Tambien fue clave
agregar el id_solicitud, ya que permite unir entrega y devolucion en un mismo
registro.

Se quitaron elementos innecesarios de algunas vistas para mejorar claridad,
se eliminaron mensajes poco institucionales y se simplificaron modales para
que el sistema fuera mas facil de usar.

El resultado final es una herramienta funcional, organizada y escalable, que
ayuda al equipo de soporte a controlar equipos, registrar evidencias y reducir
errores en el proceso de prestamo y devolucion.

============================================================
RESUMEN PARA PRESENTACION
============================================================

El proyecto consiste en un sistema web para la recepcion y devolucion de
equipos tecnologicos por sede. Se desarrollo con HTML, CSS, JavaScript y
Google Apps Script. Al inicio era un formulario de registro, pero se mejoro
hasta convertirse en un sistema completo con seleccion de sede, verificacion
de correo institucional, control de disponibilidad, firma digital, generacion
de PDFs, modulo de devolucion, cierre de solicitudes y administracion de
equipos.

Durante el proceso se agregaron mejoras visuales, validaciones, filtros,
busquedas, control por rol de soporte, cache local y documentos finales de
evidencia. Tambien se quitaron campos y botones innecesarios para dejar una
interfaz mas clara y profesional.

El principal logro fue pasar de un registro basico a un flujo completo:

ENTREGA -> USO -> DEVOLUCION -> CIERRE

Esto permite mayor trazabilidad, mejor control por sede y evidencia digital
de cada prestamo realizado.
